# ROS1 Noetic gateway lifecycle and WebUI state integration

The change adds a FastAPI-to-rospy gateway, in-process roslaunch control, periodic region commands, SSE telemetry/status delivery, and a WebUI shared by real and simulated backends. Several lifecycle boundaries report success before the underlying ROS operation is safe or complete. The highest-impact failures are concentrated in roslaunch thread ownership, blocking work on the FastAPI event loop, concurrent rospy initialization, and publication sessions that cannot invalidate work already in flight. **Watch for:** **likely** roslaunch process-monitor work running on a daemon thread; **confirmed** launch start/stop blocking all FastAPI and SSE work; **likely** overlapping launch monitors and rospy initialization; **confirmed** false-positive/stale region publication states; **confirmed** malformed geometry succeeding in simulation but failing in real ROS; **confirmed** persistence and frontend telemetry divergence.

## High-level view

The localization manager starts `ROSLaunchParent` from the Uvicorn thread but services it from a daemon monitor. **Likely:** this violates roslaunch process-monitor ownership assumptions; the shared stop event can also let an old monitor attach to a new parent, while monitor failure discards the only cleanup handle.

The localization endpoints invoke launch start and shutdown synchronously. **Confirmed:** a slow launch or shutdown pauses the sole Uvicorn worker, including SSE, region publication scheduling, and lifespan cleanup.

The rospy adapter retries startup from both the status and publication loops. **Likely:** the initialization transaction is not locked, so executor workers can create duplicate or partially leaked subscribers.

Region publication becomes active before command validation or delivery. **Confirmed:** failures leave the UI active, and stop/delete/replacement cannot invalidate an executor publication already in flight.

The region API accepts arbitrary dictionaries, while simulation returns before constructing `RegionCommand`. **Confirmed:** malformed geometry can appear successful in simulation, then publish an empty command or retry forever in real ROS.

Persistence and stream hydration have split-brain cases. **Confirmed:** failed writes leave mutated server/browser memory, late SSE clients ignore retained telemetry, and nullable telemetry becomes numeric zero in charts.

Simulation uses real-health status fields. **Confirmed:** the WebUI presents a simulated master, node, and localization launch as actual ROS infrastructure.

## roslaunch ownership crosses thread and lifecycle boundaries

**Likely:** `LocalizationManager.start()` creates and starts `ROSLaunchParent` on the Uvicorn thread, then `_monitor_loop()` calls `parent.spin_once()` from `skyforge-roslaunch-monitor`. roslaunch’s process monitor and signal/reaping behavior expect main-thread servicing; moving only `spin_once()` to a daemon thread risks unreliable child reaping and shutdown on Noetic. A supervisor process whose main thread owns `start`, `spin_once`, and `shutdown` removes this conflict (`ros_backend/localization_manager.py`, `LocalizationManager.start`, `LocalizationManager._monitor_loop`).

**Likely:** `LocalizationManager.stop()` sets a shared `_stop_event` but does not join `_monitor`. Restart clears that event after assigning a new `_parent`; an old monitor still inside its 500 ms wait can wake after the clear and service the new parent alongside the new monitor. Use a per-launch event/generation and join before restart (`ros_backend/localization_manager.py`, `LocalizationManager.start`, `LocalizationManager.stop`, `LocalizationManager._monitor_loop`).

**Likely:** when `spin_once()` raises, `_monitor_loop()` sets `_parent = None` without shutting the parent down. Later `stop()`/`close()` cannot terminate surviving children because the handle is gone. Retain the failed parent until cleanup has been attempted (`ros_backend/localization_manager.py`, `LocalizationManager._monitor_loop`, `LocalizationManager.stop`).

## Launch operations stall FastAPI and SSE

**Confirmed:** `start_localization()` and `stop_localization()` call `state.localization.start()`/`.stop()` inline, and those methods run `ROSLaunchParent.start()`/`shutdown()`. While launch processes start or terminate, the single event loop cannot serve HTTP, send SSE, or schedule publication; lifespan cleanup also calls `state.localization.close()` inline. Put roslaunch behind an asynchronously awaited supervisor that does not occupy the ASGI loop (`ros_backend/app.py`, `start_localization`, `stop_localization`, `lifespan`; `ros_backend/localization_manager.py`, `LocalizationManager.start`, `LocalizationManager.stop`).

## rospy initialization can run twice

**Likely:** `system_status_loop()` and `region_publish_loop()` both dispatch `RosBridge.ensure_started()` to the default executor. `ensure_started()` releases `_lock` after checking `_started`, then initializes the node and subscribers outside the lock. When roscore appears during active publication, two workers can overwrite `_subscribers` or leave a first subscription registered when the second constructor fails. Serialize initialization and unregister partial handles on failure (`ros_backend/app.py`, `system_status_loop`, `region_publish_loop`; `ros_backend/ros_bridge.py`, `RosBridge.ensure_started`).

## Publication state is not tied to a delivery generation

**Confirmed:** `start_publishing()` broadcasts `active: true` before ROS readiness, command construction, or first `publisher.publish()`. Repeated failures preserve `active`; the frontend stores only a boolean, hides `lastError`, and disables start. Validate/build before activation and model pending, delivering, and failed states (`ros_backend/app.py`, `start_publishing`, `region_publish_loop`; `public/app.js`, `startPublishing`, `connectEvents`, `updatePublishUi`).

**Confirmed:** `region_publish_loop()` copies the session, releases `publish_lock`, and awaits executor work. Stop, deletion, or replacement can complete while the old send remains in flight; it can publish and emit success after stop, or write its error into the new `state.publishing`. Add a generation token and check it before recording results; stop must await or invalidate the active generation (`ros_backend/app.py`, `region_publish_loop`, `start_publishing`, `stop_publishing`, `delete_region`).

## Simulation bypasses the RegionCommand contract

**Confirmed:** persistence validates only `id`, publication only checks for a dictionary, and `_region_coordinates()` turns missing geometry into an empty list while malformed values fail later. This permits zero-point commands or an active session that retries conversion errors forever. Validate shape, identifiers, polygon vertex count, finite values, and WGS84 ranges before persistence/activation (`ros_backend/app.py`, `save_region`, `start_publishing`; `ros_backend/ros_bridge.py`, `RosBridge._region_coordinates`, `RosBridge._build_region_command`; `ros1_ws/src/skyforge_msgs/msg/RegionCommand.msg`).

**Confirmed:** `RosBridge.publish_region()` returns in simulation before `_build_region_command()` runs. The documented simulation check therefore reports success for geometry that fails with real `skyforge_msgs/RegionCommand`. Build and validate the same canonical command before branching to simulated or ROS transport (`ros_backend/ros_bridge.py`, `RosBridge.publish_region`; `README.md`, “无 ROS 环境下验证 Python Gateway”).

## Region state diverges after persistence failure

**Confirmed:** save/delete mutate `state.regions` before awaiting the atomic write; an I/O failure returns an error with server memory already changed. Browser `saveRegion()` also renders optimistically and only logs POST failure. Persist a candidate copy before committing server memory, then roll back or reload browser state on failure (`ros_backend/app.py`, `save_region`, `delete_region`; `public/app.js`, `saveRegion`).

## SSE reconnect loses retained and nullable telemetry state

**Confirmed:** `GatewayState.hello()` sends `latestTopic`, but the frontend hello listener ignores it. If `/fix` arrived before connection and becomes quiet, the server retains a location while the new page remains empty. Include `latestTelemetry` or hydrate valid position data from `latestTopic.payload` (`ros_backend/app.py`, `GatewayState.hello`, `events`; `public/app.js`, `connectEvents`, `updateTelemetry`).

**Confirmed:** the backend sanitizes non-finite ROS fields to `null`, but `updateTelemetry()` applies `Number()`; JavaScript converts `null` to `0`. Unknown altitude or nullable injected fields become false zero samples. Preserve nullability and chart only finite values (`ros_backend/ros_bridge.py`, `RosBridge._finite_or_none`; `ros_backend/app.py`, `sanitize_json`; `public/app.js`, `updateTelemetry`).

## Simulation health is indistinguishable from real ROS health

**Confirmed:** simulation makes `master_online()` true, marks the adapter started, and reports localization `RUNNING` with `mock: true`. The UI ignores `settings.simulation` and `localization.mock`, displaying ROS Master `ONLINE`, a started node, and `launch active`. Render simulation as a distinct transport and localization state (`ros_backend/ros_bridge.py`, `RosBridge.master_online`, `RosBridge.ensure_started`; `ros_backend/localization_manager.py`, `LocalizationManager.start`, `LocalizationManager.status`; `public/app.js`, `renderSystemStatus`, `updateLocalizationUi`).

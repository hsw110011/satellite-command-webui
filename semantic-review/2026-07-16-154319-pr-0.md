# Independent DOM/DSM GeoTIFF upload and map interaction

The change replaces the single-map/chart layout with independent DOM and DSM viewports, browser file/folder upload, server-side tiling, embedded trajectories, and DSM elevation sampling. DOM and DSM use separate pan/zoom state, while DOM selections continue to feed region publishing. Upload bodies are streamed to temporary files rather than buffered in application memory, but map replacement does not preserve one consistent coordinate and lifecycle model. The change is not ready to ship because its required provider is outside the diff and several map/upload paths can return or display data from the wrong source.

**Watch for:** **confirmed — blocking:** `ros_backend/geotiff_provider.py` is untracked and absent from `git diff main`; **confirmed — high:** projected or rotated TIFFs use WGS84 bounding-box interpolation instead of their native affine transform; **confirmed — high:** saved DOM regions retain pixel anchors across unrelated DOM replacements; **likely — high:** concurrent uploads to one source can desynchronize the active dataset and tracked temporary path; **confirmed — high:** a 20 GiB per-request allowance has no aggregate quota or concurrency guard; **confirmed — high:** the browser creates one image node per tile and then fits the whole raster, so large dimensions remain unbounded; **confirmed — medium/security:** the API still accepts arbitrary server-local TIFF paths and returns absolute paths in metadata; **confirmed — medium:** no automated tests cover the new upload, projection, replacement, cleanup, or elevation behavior.

## High-level view

The FastAPI entry point imports a GeoTIFF provider that is not tracked. **Confirmed — blocking:** applying or committing the visible diff alone leaves the application unable to import `ros_backend.geotiff_provider` at startup.

The browser describes each raster only by WGS84 north/south/east/west bounds. **Confirmed — high:** region creation, DOM/DSM trajectory placement, and DSM pointer queries disagree with the native raster grid for projected, rotated, or skewed TIFFs; telemetry-driven ground elevation is the only new map query that starts from an already-known geographic coordinate.

Region persistence stores pixel boxes/points and identifies a map by a broad type such as `geotiff-dom`. **Confirmed — high:** replacing one DOM TIFF with another renders old regions at their previous pixels even when dimensions, bounds, or projection changed.

Upload data is written incrementally, but provider replacement and temporary-path bookkeeping are separate operations. **Likely — high:** overlapping requests can publish metadata for one upload while tiles come from another and can delete or leak the wrong temporary file.

The nominal upload ceiling does not bound system-wide resource consumption. **Confirmed — high:** each request may consume 20 GiB with no aggregate or concurrency cap. **Confirmed — high:** the frontend also materializes `tileCols × tileRows` image elements before fitting the entire raster, making large pixel dimensions a separate browser and tile-service exhaustion path.

The UI exposes only separate DOM/DSM TIFF and folder pickers, but the backend also adds `/api/map/{source}/load`. **Confirmed — medium/security:** callers can open any readable server-local file with a TIFF suffix, and metadata discloses its resolved absolute path; the default loopback bind reduces reachability but does not enforce the selector-only contract.

No test files changed with the implementation. **Confirmed — medium:** non-WGS84 transforms, map replacement, concurrent upload ordering, chunked oversize rejection, and cleanup have no regression coverage.

## Required GeoTIFF provider is outside the diff

`ros_backend/app.py` imports and constructs `GeoTiffProvider` during module import. **Confirmed — blocking:** `git status` reports `ros_backend/geotiff_provider.py` as untracked, while `git diff main` contains no file addition. A branch or patch assembled from the diff raises `ModuleNotFoundError` before FastAPI starts. Add the provider source to the change; generated `__pycache__` binaries cannot replace it.

## WGS84 bounds are not the GeoTIFF pixel transform

The provider converts the dataset envelope to WGS84 bounds, and the frontend linearly interpolates those bounds over raster width and height. **Confirmed — high:** this omits the dataset affine transform and CRS projection and is exact only for north-up, unrotated longitude/latitude rasters.

DOM rectangles and polygons therefore publish approximate coordinates; both trajectories use the inverse approximation; and a DSM pointer derives an approximate coordinate before the backend transforms it back into the DSM grid, so elevation can come from a different pixel than the cursor. Telemetry ground elevation starts with the incoming latitude/longitude, creating a behavioral inconsistency with the pointer and rendered DSM track. Return the raster georeferencing needed for exact conversion, or perform pixel/geographic conversion through rasterio-backed endpoints.

## DOM replacement reuses stale region pixels

Regions persist `pixelBox` or `pixelPoints` plus `sourceType`, and rendering accepts a region whenever its type equals the active source type. **Confirmed — high:** every uploaded DOM is labeled `geotiff-dom`, so a second TIFF renders first-TIFF regions at unchanged pixels while the old geographic bbox remains publishable. Persist a map fingerprint and hide incompatible overlays, or make geographic polygons authoritative and reproject them into the active DOM.

## Same-source uploads do not replace dataset and file atomically

`provider.load(...)` swaps the raster dataset in a worker thread; the request coroutine later updates `map_upload_paths[source]` and unlinks the previous path. **Likely — high:** two same-source requests can complete those phases in different orders, pairing the active dataset with another request's path bookkeeping. The result can be mismatched upload metadata and tiles, deletion of the active backing file on Unix, or a leaked file when unlink fails and the exception is suppressed.

Serialize validation, provider swap, path update, and deletion with a per-source lock. Keep the old provider/path pair until the new pair is validated, then switch both references as one transaction before closing and deleting the old pair.

## Large TIFFs remain unbounded after streaming

The endpoint streams chunks to disk and rejects one body after 20 GiB. **Confirmed — high:** there is no free-space reservation, aggregate byte budget, upload-count limit, or bounded concurrency, so multiple accepted requests can fill the shared temporary volume. Use a smaller configurable per-file limit, aggregate quota, free-space check, bounded concurrency, and startup cleanup for abandoned files.

`renderGeoTiff` creates an `<img>` for every tile and then fits the complete raster into the viewport. **Confirmed — high:** byte size does not constrain dimensions or tile count, so a compressible or very large TIFF can create hundreds of thousands of DOM nodes; fitting the whole image also weakens `loading="lazy"` as a request bound. Enforce dimension/tile-count limits and virtualize only tiles intersecting the current view.

## Selector-only loading is not enforced at the API boundary

The HTML removes server-path controls, but `app.py` adds `/api/map/{source}/load` with a caller-provided `path`. **Confirmed — medium/security:** this unauthenticated endpoint resolves and opens any process-readable `.tif`, `.tiff`, or `.geotiff`, and `GeoTiffProvider.metadata()` returns the resolved path. Remove the endpoint for a selector-only workflow; otherwise gate it, restrict it to an allowlisted directory, and omit absolute paths from public metadata.

## New geospatial and lifecycle paths have no regression tests

No test or spec file appears in `git diff main`. **Confirmed — medium:** independent transforms, projected/rotated conversion, region replacement, nodata/out-of-bounds elevation, streamed limits without `Content-Length`, simultaneous uploads, and temporary-file cleanup are untested. Add provider tests, API streaming/concurrency tests, and a frontend source-replacement/coordinate-rendering test.
